SELECT Lecture

SELECT est la requête la plus utilisée en SQL. Elle permet de lire des données depuis une ou plusieurs tables, avec la possibilité de filtrer, trier, grouper et paginer les résultats.

Syntaxe complète
SELECT [DISTINCT] colonnes | *
FROM table
[WHERE condition]
[GROUP BY colonne]
[HAVING condition_sur_agrégat]
[ORDER BY colonne [ASC | DESC]]
[LIMIT n OFFSET m];
// Exemples d'utilisation
📋 Toutes les colonnes d'une table
SELECT * FROM employes;
🔍 Filtrer avec WHERE et plusieurs conditions
SELECT nom, prenom, salaire FROM employes WHERE departement = 'Marketing' AND salaire BETWEEN 30000 AND 60000;
🔠 Recherche partielle avec LIKE
SELECT nom, email FROM clients WHERE email LIKE '%@gmail.com' OR nom LIKE 'Mar%';
📊 Trier, paginer et éliminer les doublons
-- 10 produits les plus chers, page 3 SELECT DISTINCT categorie, prix, nom_produit FROM produits WHERE stock > 0 ORDER BY prix DESC LIMIT 10 OFFSET 20;
🔢 Filtrer avec une liste de valeurs (IN)
SELECT nom, ville, pays FROM clients WHERE pays IN ('France', 'Belgique', 'Suisse') AND ville IS NOT NULL;
⚡
Performance — Évitez SELECT * en production. Listez uniquement les colonnes nécessaires pour réduire le trafic réseau.
🎯
HAVING vs WHERE — WHERE filtre avant l'agrégation, HAVING filtre les groupes après GROUP BY.
📄
Pagination — LIMIT n OFFSET m avec OFFSET = (page - 1) × n pour naviguer dans les résultats.
🚫
NULL — Pour tester une valeur nulle, utilisez IS NULL ou IS NOT NULL, jamais = NULL.
INSERT INTO Création

INSERT INTO ajoute une ou plusieurs nouvelles lignes dans une table. Les valeurs doivent respecter les contraintes définies (types, NOT NULL, clés étrangères…).

Syntaxe complète
-- Insertion explicite (recommandée) INSERT INTO table (col1, col2, col3) VALUES (val1, val2, val3); -- Insertion multiple (plus performant) INSERT INTO table (col1, col2) VALUES (v1, v2), (v3, v4), (v5, v6); -- Insertion depuis un SELECT INSERT INTO table (col1, col2) SELECT col1, col2 FROM autre_table;
// Exemples d'utilisation
➕ Insérer un enregistrement complet
INSERT INTO employes (nom, prenom, departement, salaire, date_embauche) VALUES ('Dupont', 'Marie', 'RH', 38000, '2024-03-15');
📦 Insérer plusieurs lignes en une seule requête
INSERT INTO produits (nom, categorie, prix, stock) VALUES ('Laptop Pro', 'Informatique', 1299.99, 50), ('Souris Sans Fil', 'Périphériques', 29.90, 200), ('Clavier Méca', 'Périphériques', 89.00, 120);
🔁 Copier des données depuis une autre table
-- Archiver les commandes de plus d'un an INSERT INTO commandes_archives (id, client_id, montant, date_cmd) SELECT id, client_id, montant, date_cmd FROM commandes WHERE date_cmd < DATE_SUB(NOW(), INTERVAL 1 YEAR);
🔄 UPSERT — Insérer ou mettre à jour si doublon
-- MySQL : ON DUPLICATE KEY UPDATE INSERT INTO statistiques (date, visites, conversions) VALUES ('2024-06-01', 1500, 45) ON DUPLICATE KEY UPDATE visites = VALUES(visites), conversions = VALUES(conversions);
📋
Colonnes explicites — Toujours lister les colonnes dans INSERT. Si la structure change, vos requêtes resteront valides.
⚡
Bulk insert — Un INSERT multi-valeurs est bien plus rapide que plusieurs INSERT individuels (moins d'allers-retours réseau).
🔑
AUTO_INCREMENT — Ne pas spécifier la colonne PK auto-incrémentée, la base de données s'en charge automatiquement.
🛡️
Injection SQL — En application, utilisez des requêtes préparées (? ou :param), jamais de concaténation de chaînes.
UPDATE Modification

UPDATE modifie les valeurs de colonnes dans des lignes existantes. Sans clause WHERE, toutes les lignes de la table sont modifiées — toujours vérifier avant d'exécuter.

Syntaxe complète
UPDATE table SET col1 = val1, col2 = val2, col3 = col3 + 1 -- calcul sur valeur existante WHERE condition; -- ⚠️ NE PAS OUBLIER !
// Exemples d'utilisation
✏️ Mettre à jour un enregistrement précis
UPDATE employes SET email = '[email protected]', telephone = '06 12 34 56 78' WHERE id = 42;
💰 Augmentation salariale conditionnelle
-- +8% pour les employés IT avec 3+ ans d'ancienneté UPDATE employes SET salaire = salaire * 1.08, derniere_augmentation = CURDATE() WHERE departement = 'IT' AND anciennete_annees >= 3;
🔁 UPDATE basé sur une sous-requête
-- Suspendre les commandes des clients inactifs UPDATE commandes SET statut = 'suspendue' WHERE client_id IN ( SELECT id FROM clients WHERE actif = 0 );
🔄 UPDATE conditionnel avec CASE
UPDATE produits SET disponibilite = CASE WHEN stock = 0 THEN 'rupture' WHEN stock < 10 THEN 'faible' ELSE 'disponible' END;
⚠️
Tester d'abord — Avant un UPDATE, exécutez un SELECT avec la même condition pour voir les lignes qui seront modifiées.
💾
Transaction — Encadrez vos UPDATE dans BEGIN … COMMIT pour pouvoir annuler avec ROLLBACK en cas d'erreur.
🔐
Vérification — Utilisez ROW_COUNT() après l'UPDATE pour vérifier le nombre de lignes réellement modifiées.
🕒
Timestamp auto — Définissez updated_at … ON UPDATE CURRENT_TIMESTAMP pour tracer automatiquement les modifications.
DELETE Suppression

DELETE supprime des lignes de façon permanente. Sans WHERE, toute la table est vidée. Pour réinitialiser une table entière, préférez TRUNCATE qui est plus rapide et remet l'auto-increment à zéro.

Syntaxe complète
DELETE FROM table WHERE condition; -- ⚠️ INDISPENSABLE ! -- Vider toute la table (plus rapide, non loggé) TRUNCATE TABLE table;
// Exemples d'utilisation
🗑️ Supprimer une ligne précise
DELETE FROM employes WHERE id = 42;
🧹 Supprimer des enregistrements anciens
-- Purger les logs de plus de 90 jours DELETE FROM logs_application WHERE created_at < DATE_SUB(NOW(), INTERVAL 90 DAY);
🔗 Supprimer avec une sous-requête
-- Commandes orphelines (client supprimé) DELETE FROM commandes WHERE client_id NOT IN ( SELECT id FROM clients );
🔄 Supprimer les doublons, garder le plus récent
DELETE FROM abonnements WHERE id NOT IN ( SELECT MAX(id) FROM abonnements GROUP BY email );
⚡ TRUNCATE — Vider et réinitialiser une table
-- Réinitialise aussi l'AUTO_INCREMENT à 1 TRUNCATE TABLE cache_sessions;
🔍
Tester avant — Exécutez SELECT * FROM table WHERE … avec la même condition pour voir ce qui sera supprimé.
↩️
ROLLBACK — DELETE dans une transaction peut être annulé. TRUNCATE est irréversible dans la plupart des SGBD.
🔗
CASCADE — Avec ON DELETE CASCADE sur une FK, supprimer un parent supprime automatiquement ses lignes enfants.
🛡️
Soft delete — En prod, préférez un flag deleted_at TIMESTAMP plutôt qu'une vraie suppression pour garder l'historique.
INNER JOIN Intersection

INNER JOIN retourne uniquement les lignes qui ont une correspondance dans les deux tables. C'est le type de jointure le plus courant — si une ligne n'a pas de correspondance de l'autre côté, elle est exclue du résultat.

Syntaxe
SELECT t1.col, t2.col
FROM table1 t1
INNER JOIN table2 t2 ON t1.id = t2.fk_id;

-- INNER est optionnel : JOIN seul = INNER JOIN FROM table1 t1 JOIN table2 t2 ON t1.id = t2.fk_id;
// Exemples d'utilisation
🔗 Commandes avec les informations client
SELECT c.nom, c.email, cmd.date_commande, cmd.montant FROM clients c INNER JOIN commandes cmd ON c.id = cmd.client_id;
🛒 Détail des lignes de commande avec nom du produit
SELECT cmd.id, p.nom_produit, lc.quantite, p.prix * lc.quantite AS sous_total FROM lignes_cmd lc INNER JOIN produits p ON lc.produit_id = p.id INNER JOIN commandes cmd ON lc.commande_id = cmd.id WHERE cmd.statut = 'confirmee';
📊 Employés et leur département (seulement si affectés)
SELECT e.nom, e.prenom, d.nom_dept, d.budget FROM employes e INNER JOIN departements d ON e.dept_id = d.id ORDER BY d.nom_dept, e.nom;
🏷️ Produits ayant au moins une commande (avec agrégat)
SELECT p.nom_produit, COUNT(lc.id) AS nb_commandes, SUM(lc.quantite) AS qte_totale FROM produits p INNER JOIN lignes_cmd lc ON p.id = lc.produit_id GROUP BY p.id ORDER BY qte_totale DESC;
🎯
Filtre implicite — INNER JOIN exclut automatiquement les lignes sans correspondance. Utile pour garantir l'intégrité des données retournées.
🔗
Jointures multiples — On peut chaîner autant de INNER JOIN que nécessaire. Chaque JOIN ajoute une table à la requête.
⚡
Index — Indexez toujours les colonnes utilisées dans les clauses ON pour de meilleures performances sur les grandes tables.
📝
Alias — Utilisez des alias courts (ex: c pour clients) pour rendre les requêtes avec plusieurs jointures plus lisibles.
LEFT JOIN Table gauche complète

LEFT JOIN retourne toutes les lignes de la table de gauche, et les lignes correspondantes de la table de droite. Si aucune correspondance n'existe à droite, les colonnes de droite sont remplies de NULL.

Syntaxe
SELECT t1.col, t2.col
FROM table1 t1
LEFT JOIN table2 t2 ON t1.id = t2.fk_id;

-- LEFT OUTER JOIN est équivalent
// Exemples d'utilisation
👥 Tous les clients, même ceux sans commande
SELECT c.nom, c.email, COUNT(cmd.id) AS nb_commandes FROM clients c LEFT JOIN commandes cmd ON c.id = cmd.client_id GROUP BY c.id;
🔍 Trouver les lignes sans correspondance (anti-join)
-- Clients n'ayant jamais passé de commande SELECT c.nom, c.email FROM clients c LEFT JOIN commandes cmd ON c.id = cmd.client_id WHERE cmd.id IS NULL;
📦 Produits avec leur catégorie (optionnelle)
SELECT p.nom_produit, p.prix, COALESCE(cat.libelle, 'Non classé') AS categorie FROM produits p LEFT JOIN categories cat ON p.categorie_id = cat.id;
📅 Employés avec leur dernière évaluation (si elle existe)
SELECT e.nom, e.prenom, ev.note, ev.date_eval FROM employes e LEFT JOIN evaluations ev ON e.id = ev.employe_id AND ev.date_eval = ( SELECT MAX(date_eval) FROM evaluations WHERE employe_id = e.id );
🚫
NULL à droite — Quand il n'y a pas de correspondance, toutes les colonnes de la table droite valent NULL. Utilisez IS NULL pour les détecter.
🔄
Anti-join — La combinaison LEFT JOIN … WHERE droite.id IS NULL est le pattern classique pour trouver les lignes sans correspondance.
🛠️
COALESCE — Utilisez COALESCE(col, valeur_defaut) pour remplacer les NULL par une valeur lisible dans les résultats.
📊
COUNT avec NULL — COUNT(*) compte les NULL, COUNT(col) les ignore. Important quand on compte des lignes de la table droite.
RIGHT JOIN Table droite complète

RIGHT JOIN est le symétrique du LEFT JOIN : toutes les lignes de la table de droite sont conservées, même sans correspondance à gauche. En pratique, un RIGHT JOIN peut toujours être réécrit en LEFT JOIN en inversant l'ordre des tables.

Syntaxe et équivalence
SELECT t1.col, t2.col
FROM table1 t1
RIGHT JOIN table2 t2 ON t1.fk_id = t2.id;

-- Équivalent avec LEFT JOIN (ordre inversé) : FROM table2 t2 LEFT JOIN table1 t1 ON t1.fk_id = t2.id;
// Exemples d'utilisation
🏢 Tous les départements, même sans employés
SELECT d.nom_dept, COUNT(e.id) AS nb_employes FROM employes e RIGHT JOIN departements d ON e.dept_id = d.id GROUP BY d.id;
📋 Toutes les catégories, avec ou sans produits
SELECT cat.libelle, COUNT(p.id) AS nb_produits FROM produits p RIGHT JOIN categories cat ON p.categorie_id = cat.id GROUP BY cat.id ORDER BY nb_produits DESC;
🔄 Réécriture en LEFT JOIN (recommandée)
-- RIGHT JOIN (moins lisible) SELECT d.nom_dept, e.nom FROM employes e RIGHT JOIN departements d ON e.dept_id = d.id; -- ✓ LEFT JOIN équivalent (préféré) SELECT d.nom_dept, e.nom FROM departements d LEFT JOIN employes e ON e.dept_id = d.id;
↩️
Préférez LEFT JOIN — La plupart des développeurs réécrivent les RIGHT JOIN en LEFT JOIN pour uniformiser le sens de lecture (de gauche à droite).
🔑
Table de référence à droite — RIGHT JOIN est utile quand la table principale (celle dont on veut toutes les lignes) est définie après un JOIN déjà écrit.
🔍
Départements vides — Cas typique : lister tous les éléments d'un référentiel (catégories, tags, statuts) même s'ils n'ont pas encore de données associées.
📊
FULL OUTER JOIN — Pour conserver toutes les lignes des deux côtés, utilisez FULL OUTER JOIN (ou UNION de LEFT + RIGHT en MySQL).
COUNT() Agrégation

COUNT() compte le nombre de lignes ou de valeurs non-NULL dans un ensemble. C'est la fonction d'agrégation la plus polyvalente, utilisable avec ou sans GROUP BY.

Variantes
COUNT(*) -- toutes les lignes (y compris NULL) COUNT(colonne) -- lignes où colonne n'est pas NULL COUNT(DISTINCT col) -- valeurs uniques non-NULL
// Exemples d'utilisation
🔢 Compter toutes les lignes d'une table
SELECT COUNT(*) AS total_employes FROM employes;
📊 Compter par groupe
SELECT departement, COUNT(*) AS nb_employes FROM employes GROUP BY departement ORDER BY nb_employes DESC;
🎯 Compter les valeurs distinctes
SELECT COUNT(*) AS total_commandes, COUNT(DISTINCT client_id) AS clients_uniques FROM commandes;
🔍 Filtrer les groupes avec HAVING
-- Départements avec plus de 10 employés SELECT departement, COUNT(*) AS nb FROM employes GROUP BY departement HAVING COUNT(*) > 10;
❗
COUNT(*) vs COUNT(col) — COUNT(*) compte toutes les lignes. COUNT(col) ignore les NULL — résultats différents si la colonne contient des NULL.
🔢
DISTINCT — COUNT(DISTINCT col) est très utile pour compter des entités uniques comme des clients ou des produits.
⚡
Performance — COUNT(*) est souvent plus rapide que COUNT(col) car il n'a pas à vérifier les NULL sur chaque ligne.
🎯
HAVING — N'utilisez jamais WHERE COUNT(*) > n : c'est invalide. Utilisez HAVING COUNT(*) > n pour filtrer après agrégation.
SUM() Agrégation

SUM() calcule la somme de toutes les valeurs numériques d'une colonne. Les valeurs NULL sont ignorées. Combinée avec GROUP BY, elle permet des totaux par catégorie.

Syntaxe
SUM(colonne_numerique)
SUM(DISTINCT colonne) -- somme des valeurs uniques
SUM(condition) -- ex: SUM(prix > 100) = nb de lignes vraies
// Exemples d'utilisation
💰 Chiffre d'affaires total
SELECT SUM(montant) AS ca_total FROM commandes WHERE statut = 'payee';
📅 CA mensuel sur l'année en cours
SELECT MONTH(date_cmd) AS mois, SUM(montant) AS ca_mensuel FROM commandes WHERE YEAR(date_cmd) = YEAR(CURDATE()) GROUP BY mois ORDER BY mois;
🏷️ Masse salariale par département
SELECT departement, SUM(salaire) AS masse_salariale, SUM(salaire) / COUNT(*) AS salaire_moyen FROM employes GROUP BY departement;
✅ Compter des conditions (astuce booléenne)
-- MySQL : TRUE = 1, FALSE = 0 SELECT SUM(salaire > 50000) AS hauts_salaires, SUM(salaire <= 50000) AS autres FROM employes;
🚫
NULL ignorés — SUM() ignore les NULL. Si toutes les valeurs sont NULL, le résultat est NULL (pas 0). Utilisez COALESCE(SUM(col), 0) si nécessaire.
💡
Astuce booléenne — SUM(condition) compte les lignes qui vérifient une condition, car TRUE vaut 1 et FALSE vaut 0 en SQL.
🔢
Types — SUM() fonctionne sur INT, DECIMAL, FLOAT… Évitez de l'utiliser sur des colonnes de type texte ou date.
📊
Totaux cumulés — Pour des sommes cumulatives, utilisez SUM() OVER (ORDER BY date) avec les fonctions de fenêtrage (window functions).
AVG() Agrégation

AVG() calcule la moyenne arithmétique d'une colonne numérique, en ignorant les valeurs NULL. Pour des moyennes pondérées ou conditionnelles, on la combine avec CASE ou des sous-requêtes.

Syntaxe
AVG(colonne) -- moyenne simple AVG(DISTINCT colonne) -- moyenne des valeurs uniques ROUND(AVG(colonne), 2) -- arrondi à 2 décimales
// Exemples d'utilisation
📊 Salaire moyen global et par département
SELECT departement, ROUND(AVG(salaire), 2) AS salaire_moyen FROM employes GROUP BY departement ORDER BY salaire_moyen DESC;
🎯 Note moyenne des produits avec filtre
SELECT p.nom_produit, ROUND(AVG(a.note), 1) AS note_moy, COUNT(a.id) AS nb_avis FROM produits p LEFT JOIN avis a ON p.id = a.produit_id GROUP BY p.id HAVING COUNT(a.id) >= 5;
🔀 Comparer à la moyenne globale
-- Employés gagnant plus que la moyenne du département SELECT e.nom, e.salaire, moy.salaire_moyen FROM employes e JOIN (SELECT departement, AVG(salaire) AS salaire_moyen FROM employes GROUP BY departement) moy ON e.departement = moy.departement WHERE e.salaire > moy.salaire_moyen;
🔢
Précision — Utilisez ROUND(AVG(col), n) pour limiter les décimales, surtout pour les montants ou pourcentages.
🚫
NULL ignorés — AVG(col) ignore les NULL, ce qui peut fausser la moyenne si les NULL représentent des zéros. Utilisez AVG(COALESCE(col, 0)) si besoin.
⚖️
Moyenne pondérée — Pour une moyenne pondérée, utilisez SUM(valeur * poids) / SUM(poids) plutôt que AVG.
🪟
Window function — AVG() OVER (PARTITION BY dept) permet de calculer la moyenne par groupe sans GROUP BY, en gardant toutes les lignes.
MIN() / MAX() Agrégation

MIN() et MAX() retournent la valeur minimale ou maximale d'une colonne. Elles fonctionnent sur les nombres, les dates et les chaînes (ordre alphabétique). Les NULL sont toujours ignorés.

Syntaxe
MIN(colonne) -- valeur la plus petite MAX(colonne) -- valeur la plus grande -- Fonctionne sur : nombres, dates, chaînes
// Exemples d'utilisation
💰 Fourchette de prix par catégorie
SELECT categorie, MIN(prix) AS prix_min, MAX(prix) AS prix_max, MAX(prix) - MIN(prix) AS ecart FROM produits GROUP BY categorie;
📅 Première et dernière commande par client
SELECT client_id, MIN(date_cmd) AS premiere_commande, MAX(date_cmd) AS derniere_commande FROM commandes GROUP BY client_id;
🔍 Trouver l'enregistrement extrême (sous-requête)
-- Produit le moins cher dans chaque catégorie SELECT * FROM produits p WHERE p.prix = ( SELECT MIN(prix) FROM produits WHERE categorie = p.categorie );
🔤 MIN/MAX sur des chaînes (ordre alphabétique)
-- Premier et dernier client alphabétiquement SELECT MIN(nom) AS premier, MAX(nom) AS dernier FROM clients;
📅
Dates — MIN(date) = date la plus ancienne, MAX(date) = date la plus récente. Très utile pour trouver la première/dernière occurrence.
🔗
Avec JOIN — Pour récupérer toute la ligne du min/max (pas juste la valeur), combinez avec une sous-requête ou un JOIN sur la valeur obtenue.
🚫
NULL ignorés — Si toutes les valeurs sont NULL, MIN/MAX retournent NULL. Pas d'erreur, mais à vérifier dans votre logique.
🪟
Window functions — MIN() OVER (PARTITION BY col) permet d'obtenir le min du groupe sur chaque ligne sans GROUP BY.
PRIMARY KEY Contrainte

La PRIMARY KEY identifie de façon unique chaque ligne d'une table. Elle combine automatiquement les contraintes NOT NULL et UNIQUE. Chaque table ne peut avoir qu'une seule clé primaire, mais elle peut porter sur plusieurs colonnes (clé composite).

Syntaxe
-- PK simple (une colonne) id INT PRIMARY KEY AUTO_INCREMENT,

-- PK composite (plusieurs colonnes) PRIMARY KEY (col1, col2),

-- Ajout après création ALTER TABLE t ADD PRIMARY KEY (id);
// Exemples d'utilisation
🔑 PK entière auto-incrémentée (cas le plus courant)
CREATE TABLE utilisateurs ( id INT PRIMARY KEY AUTO_INCREMENT, username VARCHAR(50) NOT NULL, email VARCHAR(100) UNIQUE NOT NULL );
🔗 PK composite sur une table de liaison
-- Table many-to-many : un utilisateur peut avoir plusieurs rôles CREATE TABLE utilisateurs_roles ( utilisateur_id INT NOT NULL, role_id INT NOT NULL, date_attribution DATE, PRIMARY KEY (utilisateur_id, role_id) );
🆔 PK UUID (identifiant universel)
CREATE TABLE sessions ( id CHAR(36) PRIMARY KEY DEFAULT (UUID()), user_id INT NOT NULL, expires_at DATETIME NOT NULL );
⚡
Index automatique — La PK crée automatiquement un index clusterisé (InnoDB) ou unique. Les recherches par PK sont donc extrêmement rapides.
🆔
INT vs UUID — INT AUTO_INCREMENT est plus compact et performant. UUID évite les collisions en environnement distribué mais occupe plus d'espace.
🔗
PK composite — Utilisée dans les tables de liaison many-to-many. Garantit qu'une même paire de valeurs n'apparaît pas deux fois.
⚠️
Ne jamais modifier — Changer la valeur d'une PK peut casser toutes les clés étrangères qui la référencent. Concevez-la stable dès le départ.
FOREIGN KEY Contrainte

La FOREIGN KEY crée un lien entre deux tables en référençant la clé primaire d'une autre table. Elle garantit l'intégrité référentielle : impossible d'insérer une valeur qui n'existe pas dans la table parente.

Syntaxe avec actions en cascade
FOREIGN KEY (col_locale)
REFERENCES table_parente(col_pk)
ON DELETE [CASCADE | SET NULL | RESTRICT]
ON UPDATE [CASCADE | SET NULL | RESTRICT]
// Exemples d'utilisation
🔗 Relation simple commandes → clients
CREATE TABLE commandes ( id INT PRIMARY KEY AUTO_INCREMENT, client_id INT NOT NULL, montant DECIMAL(10,2), FOREIGN KEY (client_id) REFERENCES clients(id) ON DELETE RESTRICT -- empêche la suppression d'un client ayant des commandes );
🗑️ Suppression en cascade (ON DELETE CASCADE)
CREATE TABLE commentaires ( id INT PRIMARY KEY AUTO_INCREMENT, post_id INT NOT NULL, contenu TEXT, FOREIGN KEY (post_id) REFERENCES posts(id) ON DELETE CASCADE -- supprime les commentaires si le post est supprimé );
🔄 Mise à null (ON DELETE SET NULL)
CREATE TABLE taches ( id INT PRIMARY KEY AUTO_INCREMENT, assignee_id INT, -- peut être NULL (non assignée) titre VARCHAR(200), FOREIGN KEY (assignee_id) REFERENCES employes(id) ON DELETE SET NULL -- tâche devient non-assignée si l'employé part );
🛡️
Intégrité — La FK empêche d'insérer une commande pour un client inexistant, ou de supprimer un client qui a des commandes (RESTRICT).
⚡
Index recommandé — Indexez toujours la colonne FK. MySQL/InnoDB le fait automatiquement, mais d'autres SGBD ne le font pas.
🔄
CASCADE vs RESTRICT — CASCADE propage automatiquement, RESTRICT bloque. Choisissez selon le domaine métier : CASCADE pour les données dépendantes, RESTRICT pour les données importantes.
🗂️
Ordre de création — Créez toujours la table parente avant la table enfant. Pour supprimer, faites l'inverse (enfant d'abord).
UNIQUE Contrainte

La contrainte UNIQUE garantit que toutes les valeurs d'une colonne (ou combinaison de colonnes) sont différentes. Contrairement à la PRIMARY KEY, une colonne UNIQUE peut contenir des valeurs NULL (plusieurs NULL sont tolérés dans la plupart des SGBD).

Syntaxe
-- Sur une colonne email VARCHAR(100) UNIQUE,

-- UNIQUE composite (combinaison unique) CONSTRAINT uq_nom UNIQUE (col1, col2),

-- Ajout après création ALTER TABLE t ADD UNIQUE (email);
// Exemples d'utilisation
📧 Email unique par utilisateur
CREATE TABLE utilisateurs ( id INT PRIMARY KEY AUTO_INCREMENT, email VARCHAR(100) UNIQUE NOT NULL, nom VARCHAR(50) NOT NULL );
🔑 Contrainte UNIQUE composite
-- Un utilisateur ne peut s'abonner qu'une fois à une newsletter CREATE TABLE abonnements ( user_id INT, newsletter_id INT, date_sub DATE, CONSTRAINT uq_user_newsletter UNIQUE (user_id, newsletter_id) );
🔄 INSERT avec gestion de doublon (ON DUPLICATE KEY)
-- Utiliser la contrainte UNIQUE pour faire un UPSERT INSERT INTO profils (user_id, bio, avatar_url) VALUES (42, 'Développeur passionné', '/img/42.jpg') ON DUPLICATE KEY UPDATE bio = VALUES(bio), avatar_url = VALUES(avatar_url);
🆚
UNIQUE vs PRIMARY KEY — Une table peut avoir plusieurs contraintes UNIQUE, mais une seule PK. UNIQUE accepte NULL, PK non.
⚡
Index automatique — Chaque contrainte UNIQUE crée automatiquement un index, ce qui accélère les recherches sur cette colonne.
🔢
NULL et UNIQUE — La plupart des SGBD (MySQL, PostgreSQL) autorisent plusieurs NULL dans une colonne UNIQUE car NULL ≠ NULL.
💡
UPSERT — La contrainte UNIQUE est souvent utilisée avec ON DUPLICATE KEY UPDATE (MySQL) ou ON CONFLICT DO UPDATE (PostgreSQL) pour faire des insertions conditionnelles.
NOT NULL Contrainte

La contrainte NOT NULL garantit qu'une colonne doit toujours contenir une valeur. Si aucune valeur n'est fournie lors d'un INSERT, la base retourne une erreur. C'est la contrainte la plus simple mais aussi l'une des plus importantes pour la qualité des données.

Syntaxe
-- À la création nom VARCHAR(50) NOT NULL,

-- Avec valeur par défaut statut VARCHAR(20) NOT NULL DEFAULT 'actif',

-- Modifier une colonne existante ALTER TABLE t MODIFY nom VARCHAR(50) NOT NULL;
// Exemples d'utilisation
📋 Table avec colonnes obligatoires et optionnelles
CREATE TABLE employes ( id INT PRIMARY KEY AUTO_INCREMENT, nom VARCHAR(50) NOT NULL, -- obligatoire prenom VARCHAR(50) NOT NULL, -- obligatoire telephone VARCHAR(20), -- optionnel (NULL ok) date_depart DATE -- NULL = encore en poste );
⚙️ NOT NULL avec DEFAULT pour les champs techniques
CREATE TABLE articles ( id INT PRIMARY KEY AUTO_INCREMENT, titre VARCHAR(200) NOT NULL, statut ENUM('brouillon','publie') NOT NULL DEFAULT 'brouillon', created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP );
🔍 Trouver les lignes avec des valeurs NULL
-- Contrôle qualité : lignes avec champs vides SELECT * FROM clients WHERE email IS NULL OR telephone IS NULL OR adresse IS NULL;
🧮
NULL en calcul — Toute opération arithmétique avec NULL retourne NULL. Ex: 5 + NULL = NULL. Utilisez COALESCE(col, 0) pour les totaux.
⚙️
DEFAULT — Combinez NOT NULL avec DEFAULT pour éviter les erreurs lors des INSERT qui n'incluent pas cette colonne.
📊
Sémantique NULL — NULL signifie "valeur inconnue ou absente". Parfois un NULL intentionnel est utile (ex: date_depart = NULL = encore en poste).
🔍
Comparaison NULL — col = NULL retourne toujours FAUX. Utilisez toujours col IS NULL ou col IS NOT NULL pour tester les NULL.
1NF Première forme normale

La Première Forme Normale (1NF) exige que chaque cellule contienne une valeur atomique (indivisible) et qu'il n'y ait pas de groupes répétitifs. C'est le prérequis de toute base de données relationnelle.

Règles de la 1NF
1. Chaque cellule contient une seule valeur (atomique)
2. Pas de colonnes répétées (produit1, produit2, produit3…)
3. Pas de listes dans une cellule ("rouge, bleu, vert")
4. Chaque ligne est unique (une PK doit exister)
// Exemples : avant / après 1NF
❌ Violation 1NF — Liste dans une cellule
-- commandes(id, client, produits) -- id=1, 'Alice', 'Laptop, Souris, Clavier' ← INVALIDE -- Impossible de requêter un produit spécifique
❌ Violation 1NF — Colonnes répétées
-- commandes(id, client, produit1, produit2, produit3) -- Que faire si le client commande 4 produits ?
✅ Conforme 1NF — Une ligne par valeur
-- Table de liaison : une ligne = un produit d'une commande CREATE TABLE lignes_commande ( commande_id INT NOT NULL, produit_id INT NOT NULL, quantite INT NOT NULL, PRIMARY KEY (commande_id, produit_id) ); -- ✓ Chaque cellule = 1 valeur, recherche possible
🔑
Identifiant unique — Toute table 1NF doit avoir une PRIMARY KEY qui identifie chaque ligne de façon unique.
💡
Colonnes répétées — Si vous avez produit1, produit2, produit3… c'est un signal fort que vous avez besoin d'une table séparée avec une FK.
📋
JSON/Arrays — Stocker du JSON dans une colonne peut violer la 1NF. Acceptable pour des données flexibles, mais au coût de la queryabilité SQL.
🧱
Fondation — Sans 1NF, les autres formes normales (2NF, 3NF) ne peuvent pas s'appliquer. C'est la brique de base.
2NF Deuxième forme normale

La Deuxième Forme Normale (2NF) exige que la table soit déjà en 1NF et que tous les attributs non-clés dépendent de la totalité de la clé primaire (pas seulement d'une partie). Elle ne s'applique que si la PK est composite.

Règle de la 2NF
1. La table est en 1NF
2. Tout attribut non-clé dépend de toute la PK
→ Pas de dépendance partielle
→ Si PK = (A, B), chaque col doit dépendre de (A ET B)
// Exemples : avant / après 2NF
❌ Violation 2NF — Dépendance partielle
-- PK composite : (commande_id, produit_id) -- commandes_produits(commande_id, produit_id, quantite, nom_produit, prix_produit) -- ❌ nom_produit et prix_produit dépendent SEULEMENT de produit_id -- pas de la paire (commande_id, produit_id) → redondance !
✅ Conforme 2NF — Séparation des dépendances
-- Table 1 : infos produit (dépendent de produit_id seul) CREATE TABLE produits ( id INT PRIMARY KEY, nom VARCHAR(100), prix DECIMAL(8,2) ); -- Table 2 : lignes de commande (quantite dépend de (cmd_id, prod_id)) CREATE TABLE lignes_commande ( commande_id INT, produit_id INT, quantite INT, PRIMARY KEY (commande_id, produit_id) );
💡 Identifier une dépendance partielle
-- Question à se poser pour chaque colonne : -- "Cette valeur change-t-elle si je change UNE SEULE partie de la PK ?" -- Si OUI → dépendance partielle → violation 2NF → extraire dans une table
🔑
PK simple — Si votre PK est sur une seule colonne, votre table est automatiquement en 2NF (pas de dépendance partielle possible).
🔄
Redondance — La violation 2NF cause des redondances : le nom d'un produit stocké dans chaque ligne de commande. Un UPDATE raté crée des incohérences.
✂️
Solution — Extraire les colonnes partiellement dépendantes dans leur propre table, et référencer avec une FK.
📊
Tables many-to-many — Les tables de liaison avec PK composite sont les candidates typiques à la vérification 2NF.
3NF Troisième forme normale

La Troisième Forme Normale (3NF) exige que la table soit en 2NF et qu'aucun attribut non-clé ne dépende d'un autre attribut non-clé (pas de dépendance transitive). C'est la cible standard en conception de bases relationnelles.

Règle de la 3NF
1. La table est en 2NF
2. Aucune dépendance transitive :
A → B → C (si C dépend de B qui dépend de A)
alors C ne doit pas être dans la même table que A
// Exemples : avant / après 3NF
❌ Violation 3NF — Dépendance transitive
-- employes(id, nom, code_postal, ville, region) -- PK = id -- ❌ ville dépend de code_postal, pas directement de id -- ❌ region dépend de ville → chaîne : id → code_postal → ville → region
✅ Conforme 3NF — Extraction des dépendances transitives
CREATE TABLE codes_postaux ( code_postal CHAR(5) PRIMARY KEY, ville VARCHAR(100), region VARCHAR(50) ); CREATE TABLE employes ( id INT PRIMARY KEY, nom VARCHAR(50), code_postal CHAR(5) REFERENCES codes_postaux(code_postal) -- ville et region sont dans codes_postaux → plus de redondance );
💡 Autre exemple : commandes et infos client
-- ❌ commandes(id, client_id, client_nom, client_email, montant) -- client_nom et client_email dépendent de client_id, pas de l'id commande -- ✓ Solution 3NF CREATE TABLE clients (id, nom, email); CREATE TABLE commandes (id, client_id REFERENCES clients(id), montant);
🔗
Dépendance transitive — Si A→B et B→C, alors A→C de façon transitive. C doit être extrait dans la table de B, pas dans celle de A.
♻️
Éviter la redondance — Sans 3NF, mettre à jour l'email d'un client oblige à modifier toutes ses commandes. Avec 3NF, une seule modification suffit.
⚖️
Dénormalisation — En prod, on accepte parfois des violations 3NF (colonnes calculées, copies) pour des raisons de performance. C'est un choix délibéré.
🎯
Objectif 3NF — Une base en 3NF minimise la redondance et les anomalies de mise à jour. C'est la cible de conception pour la majorité des applications.
